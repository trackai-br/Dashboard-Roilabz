# Story 1.4: RLS Policies — Row Level Security

## 📌 Overview

**Epic:** 1 — Core Sync & Dashboard Básico
**Story ID:** 1.4
**Complexity:** Medium (M)
**Estimated Points:** 8
**Status:** 🟡 Draft

---

## 📖 User Story

> Como **Account Manager**, quero que **minha conta e dados sejam isolados** de outros usuários, de forma que **RLS no banco de dados impeça acesso não autorizado**.

---

## ✅ Acceptance Criteria

- [ ] Tabelas de usuários criadas: `users`, `user_account_access`, `access_logs`
- [ ] RLS habilitado em todas as tabelas sensíveis
- [ ] RLS policies: usuários só veem suas próprias contas
- [ ] RLS policies: admins de conta veem todos os acessos da conta
- [ ] Tabela access_logs registra todas as ações (audit trail)
- [ ] Índices criados para performance
- [ ] API `/accounts` respeta RLS automaticamente
- [ ] Testes: RLS bloqueia acesso não autorizado

---

## 🎯 Implementation Plan

### Phase 1: Database Schema (3 points)

- [ ] Migration 002: Criar tabelas `users`, `user_account_access`, `access_logs`
- [ ] Adicionar índices (user_id, account_id, created_at)
- [ ] Criar triggers para timestamps automáticos (updated_at)

**Acceptance:**
- Tabelas criadas com colunas corretas
- Índices existem e melhoram query performance
- Foreign keys configuradas corretamente

### Phase 2: RLS Policies (3 points)

- [ ] RLS policies para tabela `users`: usuários leem/atualizam a si mesmos
- [ ] RLS policies para `user_account_access`: usuários veem seus próprios acessos
- [ ] RLS policies para `access_logs`: usuários veem logs de suas contas
- [ ] RLS policies para `meta_accounts`: usuários veem apenas contas às quais têm acesso
- [ ] Service role bypassa RLS (para Inngest sync)

**Acceptance:**
- Policies codificadas corretamente (SELECT, INSERT, UPDATE restrictions)
- User-based policies funcionam (auth.uid() matching)
- Admin-based policies funcionam (role = 'admin' checking)
- Service role consegue fazer insert/update (para sync)

### Phase 3: Helpers & API (2 points)

- [ ] Criar `src/lib/supabase-rls.ts` com helpers: `getUserAccounts()`, `hasAccountAccess()`, `logAccess()`
- [ ] Criar `src/pages/api/accounts/index.ts` que retorna contas do usuário (RLS enforced)
- [ ] Testes de RLS enforcement

**Acceptance:**
- API `/accounts` retorna apenas contas do usuário logado
- Tentativa de acesso cross-user é bloqueada pelo RLS
- Access logs são registrados automaticamente

## 📝 Notes & Considerations

### Security-First Approach
- RLS é a primeira linha de defesa contra acesso não autorizado
- Todas as policies começam com DENY implícito
- Service role é usado apenas para operações do sistema (Inngest)

### Performance Considerations
- Índices em user_id, account_id, created_at
- Access logs podem crescer (considerar partição no futuro)
- Queries com RLS são otimizadas pelo Supabase

### Migration Strategy
- Migration é executada via Supabase SQL Editor
- Rollback: Drop policies, disable RLS, drop tables
- Replicar RLS em ambiente de staging antes de prod
